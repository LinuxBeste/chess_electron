# Tournament System

Tournaments use single-elimination brackets with automatic seeding and match progression.

## Architecture

```
Create tournament  ──→  Players join  ──→  Start tournament  ──→  Rounds advance
      │                     │                     │                       │
  POST /tournaments    POST /tournaments/:id/join       idempotent    auto-detected
  (creates row in     (adds participant row,         via POST          on game
   tournaments +       increments participants       /tournaments/     completion
   joins creator)      JSONB field)                  :id/start
```

## Tournament lifecycle

### States

| Status      | Description                               |
| ----------- | ----------------------------------------- |
| `waiting`   | Open for registration. Players can join.  |
| `active`    | Bracket locked. Matches in progress.      |
| `completed` | All matches finished. Winner recorded.    |
| `cancelled` | Deleted (removes participants + matches). |

### 1. Create

**REST:**

```http
POST /tournaments
Authorization: Bearer <token>
Content-Type: application/json

{ "name": "Chess Cup", "maxPlayers": 8, "isPrivate": false }
```

Response:

```json
{
  "id": "uuid",
  "name": "Chess Cup",
  "status": "waiting",
  "maxPlayers": 8,
  "isPrivate": false,
  "joinCode": null,
  "createdAt": 1700000000000
}
```

Private tournaments return a random 8-character `joinCode` (uppercase alphanumeric):

```json
{ "joinCode": "A3F7K2P9" }
```

**DB:** Creates row in `tournaments` and adds creator as first participant.

### 2. Join

**Public tournament:**

```http
POST /tournaments/:id/join
Authorization: Bearer <token>
```

**Private tournament by code:**

```http
POST /tournaments/join-by-code/:code
Authorization: Bearer <token>
```

Conditions:

- Tournament must be in `waiting` status.
- Player must be registered (not anonymous).
- Player must not already be a participant.
- Max players not yet reached.

**DB:** Adds row in `tournament_participants`. Updates `participant_data` JSONB on `tournaments` row.

### 3. Start

```http
POST /tournaments/:id/start
Authorization: Bearer <token>  (must be creator)
```

**Seeding algorithm:**

1. Participants are shuffled randomly.
2. First-round pairings are created: (0 vs N-1), (1 vs N-2), etc. (standard bracket seeding).
3. Byes (empty slots when participants < maxPlayers) result in automatic wins.
4. Matches are created in `tournament_matches` table with `round=1`.

**Round 1 example (8-player bracket after shuffle):**

| Position | White  | Black  |
| -------- | ------ | ------ |
| 0        | Seed 0 | Seed 7 |
| 1        | Seed 3 | Seed 4 |
| 2        | Seed 1 | Seed 6 |
| 3        | Seed 2 | Seed 5 |

### 4. Play matches

When a tournament match is started (first move), a game is created and `game_id` is linked. During the game, a game-level clock is displayed. The result updates the match:

- Winner advances to the next round.
- Loser is eliminated.

**Auto-advance:** When all matches in a round are completed, the next round is automatically generated by the server (no explicit trigger needed — checked on game completion).

### 5. Completion

When only one player remains (or the final match is decided):

- Tournament status → `completed`
- `winner_id` set
- `completed_at` timestamp set

## REST API

| Method   | Endpoint                          | Auth | Description                           |
| -------- | --------------------------------- | ---- | ------------------------------------- |
| `GET`    | `/tournaments`                    | -    | List public tournaments               |
| `GET`    | `/tournaments?status=waiting`     | -    | Filter by status                      |
| `GET`    | `/tournaments/:id`                | -    | Tournament details with participants  |
| `POST`   | `/tournaments`                    | ✓    | Create tournament                     |
| `POST`   | `/tournaments/:id/join`           | ✓    | Join public tournament                |
| `POST`   | `/tournaments/:id/start`          | ✓    | Start tournament (creator only)       |
| `POST`   | `/tournaments/join-by-code/:code` | ✓    | Join private tournament               |
| `PUT`    | `/tournaments/:id`                | ✓    | Update name/maxPlayers (creator only) |
| `DELETE` | `/tournaments/:id`                | ✓    | Cancel tournament (creator only)      |

## Database tables

### `tournaments`

| Column             | Type    | Description                                   |
| ------------------ | ------- | --------------------------------------------- |
| `id`               | TEXT PK | UUID v4                                       |
| `name`             | TEXT    | Tournament name                               |
| `status`           | TEXT    | `waiting`, `active`, `completed`, `cancelled` |
| `created_by`       | TEXT FK | Creator's user ID                             |
| `max_players`      | INTEGER | Player cap (default 8)                        |
| `is_private`       | BOOLEAN | Hidden from public list                       |
| `join_code`        | TEXT    | 8-char code for private tournaments           |
| `type`             | TEXT    | Always `single_elimination`                   |
| `participant_data` | JSONB   | Denormalized participant list                 |
| `match_data`       | JSONB   | Denormalized match list                       |
| `winner_id`        | TEXT FK | Winner's user ID                              |

### `tournament_participants`

| Column          | Type    | Description               |
| --------------- | ------- | ------------------------- |
| `id`            | TEXT PK | UUID v4                   |
| `tournament_id` | TEXT FK | Parent tournament         |
| `player_id`     | TEXT FK | Participant's user ID     |
| `display_name`  | TEXT    | Denormalized display name |
| `seed`          | INTEGER | Bracket seed position     |

### `tournament_matches`

| Column            | Type    | Description                      |
| ----------------- | ------- | -------------------------------- |
| `id`              | TEXT PK | UUID v4                          |
| `tournament_id`   | TEXT FK | Parent tournament                |
| `round`           | INTEGER | Round number (1=first)           |
| `position`        | INTEGER | Position within round            |
| `white_player_id` | TEXT    | White player (nullable for bye)  |
| `black_player_id` | TEXT    | Black player (nullable for bye)  |
| `game_id`         | TEXT    | Game UUID once match starts      |
| `winner_id`       | TEXT    | Winner UUID                      |
| `status`          | TEXT    | `pending`, `active`, `completed` |

## WebSocket events

| Event                | Trigger                | Payload                      |
| -------------------- | ---------------------- | ---------------------------- |
| `tournament_created` | New tournament created | `{ tournamentId, name }`     |
| `tournament_joined`  | Player joins           | `{ tournamentId, playerId }` |
| `tournament_started` | Tournament begins      | `{ tournamentId }`           |

These are broadcast to relevant players (currently limited — WebSocket integration for tournaments is minimal; most data is fetched via REST).

## Bracket visualization

The admin dashboard renders brackets from `tournament_matches`, grouped by `round`. Each match shows:

- White player name (or "TBD")
- Black player name (or "TBD")
- Result or "in progress"
- Game link (if active)

The front-end computes bracket layout from the round/position structure.

---

## Limitations

- **Single elimination only** — no round-robin or Swiss system.
- **No draw handling** — draws must be resolved by the players (play until decisive result).
- **No time control enforcement** — tournament games use whatever time control the creator set on the game.
- **No automatic rematch** — draws require manual replay or admin intervention.
- **Max players** must be a power of 2 (4, 8, 16, 32) for clean bracket seeding. Non-power-of-2 will create byes.
